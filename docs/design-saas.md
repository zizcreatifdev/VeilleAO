## Problem Statement

Les agences de communication et de branding au Sénégal ratent des appels d'offres
publics faute d'un système de veille fiable. Le scraper Veille AO V1 (déjà livré)
envoie un digest email quotidien à une seule adresse. Le produit SaaS transforme
cette infrastructure en service multi-agences payant — chaque agence obtient son
propre accès filtré, son équipe, et ses alertes personnalisées.

---

## Demand Evidence

- Demande réelle confirmée : client nommé (agence spécifique + contact identifié)
- V1 du scraper opérationnel : preuve que la donnée est disponible et utilisable
- Paiement Wave anticipé jugé acceptable par le client cible
- Marché : agences com/branding Sénégal

---

## Status Quo

Les agences font la veille AO manuellement :
- Consultent DCMP, PNUD, marchesdusenegal.com, senoffre.com, joffres.net séparément
- 2-4h/semaine de travail répétitif par agence
- Alertes manquées si la personne responsable est absente
- Pas de filtrage par secteur — tout doit être lu

---

## Target User & Narrowest Wedge

**User principal :** Directeur ou responsable développement commercial d'une agence
de communication dakaroise (5-30 personnes). Travaille sur desktop pour les AOs
(documents PDF, soumissions formelles). Accède aux alertes depuis mobile pendant
les déplacements.

**Wedge (semaine 1) :** Landing page + paiement Wave anticipé. L'agence paie,
reçoit une confirmation par email, et est inscrite sur la liste des premiers
accès. En attendant le dashboard (semaine 3-5), elles reçoivent le digest
email quotidien du scraper V1 directement.

---

## Contraintes

- **Marché Wave-first** : pas de Stripe, pas de carte bancaire. Wave Business + confirmation manuelle = V1. API webhook Wave en Phase 2.
- **Responsive uniquement** : pas de PWA manifest en V1. Valider l'usage mobile vs desktop sur les 3 premiers clients avant d'investir en PWA.
- **Budget infra** : Supabase Pro ($25/mois dès le début — le free tier pause les projets inactifs). Vercel gratuit, Clerk gratuit jusqu'à 10k MAU, Resend gratuit jusqu'à 100 emails/jour.
- **Équipe** : solo founder + Claude Code.

---

## Prémisses validées

1. Landing page + paiement anticipé = wedge correct pour valider cette semaine
2. Wave semi-auto (confirmation manuelle) est suffisant pour V1 — acceptable par le marché cible
3. Clerk est le bon choix auth pour la gestion multi-tenant (Plan Studio 2 users / Plan Premium 10 users)
4. Architecture 3 niveaux nécessaire dès V1 : Admin / Agence / Landing — livrée en séquence
5. ~~PWA mobile-first~~ → **Responsive uniquement** — révisé après challenge du sous-agent

---

## Avis Cross-Model

Le sous-agent Claude a challengé la prémisse PWA (révisée en "responsive only")
et a proposé un prototype 48h radicalement minimal. L'insight clé :
*"Le scraper envoie déjà un digest à une adresse — c'est le produit."*

**Décision architecturale résultante :** Livraison séquencée (Lean), architecture
extensible (Full SaaS) — Clerk + schéma Supabase multi-tenant installés dès le départ
même si un seul user par agence en V1, pour éviter toute migration future.

## Architecture Technique

### Stack

| Couche | Choix | Raison |
|---|---|---|
| Frontend | Next.js 15 App Router | SSR + Server Actions pour les mutations Wave |
| Deployment | Vercel | Zero-config, preview branches |
| Auth + Multi-tenant | Clerk (Organizations) | Seat limits natifs par plan |
| Database | Supabase Pro PostgreSQL + RLS | $25/mois, pas de pause automatique |
| Email | Resend + React Email | Digest quotidien + emails transactionnels |
| Payment | Wave semi-auto | Lien Wave pré-rempli + ref T_[A-Z0-9]{18} + confirmation admin |
| Scraper | Python V1 existant | GitHub Actions cron 07h UTC — inchangé en V1 |

### Schéma Supabase (tables clés)

```sql
-- Inscriptions liste d'attente (AVANT la Clerk Org — pont identité waitlist → actif)
waitlist_signups (
  id uuid PRIMARY KEY,
  email text UNIQUE,               -- lien vers l'agence future
  agency_name text,
  phone text,
  plan text CHECK (plan IN ('studio', 'premium')),
  created_at timestamptz
)
-- À l'activation Wave : waitlist_signups.email → agencies.notification_email + Clerk Org

-- Agences (= Clerk Organization, créé à l'activation du paiement)
agencies (
  id uuid PRIMARY KEY,
  clerk_org_id text UNIQUE,        -- Clerk Org ID, null jusqu'à l'activation
  waitlist_signup_id uuid REFERENCES waitlist_signups,
  name text,
  plan text CHECK (plan IN ('studio', 'premium')),
  status text CHECK (status IN ('waitlist', 'active', 'suspended')),
  notification_email text,         -- email de réception du digest (V1 compat + Phase 2)
  created_at timestamptz
)

-- Paiements Wave
payments (
  id uuid PRIMARY KEY,
  agency_id uuid REFERENCES agencies,
  montant_net integer,             -- FCFA hors frais Wave (montant affiché)
  montant_brut integer,            -- FCFA facturé (montant_net × 1.01, arrondi)
  statut text CHECK (statut IN ('en_attente', 'en_attente_confirmation', 'confirme', 'annule')),
  reference_wave text,             -- T_[A-Z0-9]{18}
  numero_facture text,             -- FAC-2026-ZC-0001 (Postgres sequence)
  numero_recu text,                -- REC-2026-ZC-0001 (Postgres sequence)
  created_at timestamptz
)

-- AOs partagés (mêmes données pour toutes les agences — pas de RLS sur cette table)
appels_offres (
  id uuid PRIMARY KEY,
  url_hash text UNIQUE,            -- SHA256 de l'URL normalisée — dedup global
  title text,
  url text,
  source text,
  deadline text,
  description text,
  language text,
  scraped_at timestamptz
)
-- Note sécurité : appels_offres a une RLS "read-all" (pas d'isolation par org).
-- Le filtrage par keywords est application-enforced dans la query dashboard,
-- PAS via RLS. Un bug de query peut exposer tous les AOs — à tester explicitement.

-- Préférences par agence (keywords supplémentaires)
org_preferences (
  id uuid PRIMARY KEY,
  agency_id uuid REFERENCES agencies,
  extra_keywords text[]            -- en plus des KEYWORDS_GLOBAUX constants
  -- Digest envoyé à agencies.notification_email (tous les membres ne reçoivent
  -- pas un email individuel en V1 — un seul email par org)
)

-- AOs vus par agence (pour le badge "nouveau" et le statut)
org_seen_aos (
  agency_id uuid REFERENCES agencies,
  ao_id uuid REFERENCES appels_offres,
  seen_at timestamptz,
  status text CHECK (status IN ('interesting', 'ignored', 'applied')),
  PRIMARY KEY (agency_id, ao_id)
)
-- RLS sur org_preferences et org_seen_aos : Clerk org_id via JWT claim
```

**Keywords globaux :** définis comme constante dans le code Next.js
(`lib/keywords.ts` — même liste que `scraper/config.py`). Pas de table
`global_keywords` en V1 — les modifier nécessite un deploy. Phase 2 :
table `params` pour les rendre éditables depuis l'admin dashboard.

**Numéros de facture/reçu :** Postgres sequence `invoice_seq` → format
`FAC-{year}-ZC-{seq::5}`. Race-condition-safe par design natif Postgres.

### Multi-tenant avec Clerk

```
Admin (toi, le builder) → Clerk User avec publicMetadata { role: 'owner' }
  → Auth admin dashboard : middleware vérifie publicMetadata.role === 'owner'
  → L'admin crée les Clerk Orgs via Clerk Management API (server-side)
    — PAS manuellement via le dashboard Clerk

Agence → Clerk Organization (créée server-side lors de la confirmation Wave)
  Trigger dans l'API PATCH /api/admin/payments/[id] :
    const org = await clerkClient.organizations.createOrganization({
      name: agency.name,
      maxAllowedMemberships: plan === 'studio' ? 2 : 10,
    })
    await clerkClient.organizations.updateOrganization(org.id, {
      publicMetadata: { agency_id: agency.id, plan }
    })
  └── Admin agence → OrganizationMember (role: 'org:admin')
      — invitation Clerk envoyée à agencies.notification_email
  └── Users agence → OrganizationMember (role: 'org:member')

Plan Studio  → maxAllowedMemberships: 2  (enforced par Clerk)
Plan Premium → maxAllowedMemberships: 10

Dépassement Studio : Clerk API retourne 403 → UI affiche "Passez au plan Premium."
```

**Emails transactionnels (Resend) :**
1. Confirmation inscription waitlist (automatique à la soumission du formulaire)
2. Confirmation Wave reçue (après soumission T_[A-Z0-9]{18})
3. Activation compte (après confirmation admin) + **reçu HTML par email** (V1)
4. Invitation membre (Clerk natif)
5. Relance Wave non confirmée (Resend, déclenché après 48h)

**PDF (reçu/facture) :** Différé en Phase 2. En V1, le reçu est un email HTML formaté
avec Resend + React Email — suffisant pour 1-10 clients, sans les risques de bundle size
de `@react-pdf/renderer` sur Vercel serverless.

**Note :** Clerk Organizations n'est activé qu'en semaine 3 (dashboard agence).
En semaine 1-2, auth admin = simple Clerk User. L'org est créée server-side
via Clerk Management API au moment de la confirmation Wave (step 7 du flow).

### Flow Paiement Wave — détail complet

```
1. Client choisit le plan (Studio ou Premium) sur la landing page
2. Prix affiché clairement en FCFA (ex: 25 000 FCFA/mois)
3. Bouton "Payer via Wave" → lien Wave pré-rempli :
   montantAvecFrais = Math.round(montant * 1.01)
   url = `${WAVE_BASE_URL}?amount=${montantAvecFrais}`
4. Client paie depuis son app Wave Business
5. Wave génère une référence : T_[A-Z0-9]{18} (ex: T_TYKJGIPX5K6SMO3W)
6. Client soumet la référence sur la page de confirmation
   → Validation regex /^T_[A-Z0-9]{18}$/ côté client ET serveur
   → Insert en DB : payments.statut = 'en_attente_confirmation'
7. Email auto à l'admin :
   "Nouvelle référence Wave — [Nom agence] [Email] [Plan] [Montant FCFA] [T_...]"
8. Admin vérifie dans son dashboard Wave Business
9. Admin clique "Confirmer" dans l'admin dashboard (bouton unique)
   → payments.statut = 'confirme'
   → agencies.status = 'active'
   → Clerk Organization créée avec maxAllowedMemberships selon le plan
   → Email client : confirmation + lien dashboard + PDF reçu en pièce jointe
10. Si Wave non confirmé après 48h → email de relance automatique (Resend)

SLA : confirmations traitées les jours ouvrés (lun-ven).
La landing page mentionne explicitement "activation sous 24h jours ouvrés."
```

**PDF généré avec `@react-pdf/renderer` :**
- Facture : FAC-2026-ZC-XXXX — logo, agence, plan, montant FCFA, date
- Reçu : REC-2026-ZC-XXXX — même contenu + mention "Paiement reçu" + référence Wave

### Pipeline Scraper → Dashboard

```
GitHub Actions cron 07h UTC (inchangé)
  → scraper Python écrit dans Supabase via service-role key :
    upsert appels_offres ON CONFLICT (url_hash) DO NOTHING
  → Dashboard agence lit appels_offres filtré par :
    (keywords globaux config.py) UNION (org_preferences.extra_keywords)
    et LEFT JOIN org_seen_aos pour le badge "nouveau"
```

**Risque GitHub Actions :** Le cron GH Actions peut être retardé ou throttlé.
Acceptable pour ≤10 clients. Si la fiabilité devient un problème après le lancement,
migration vers Vercel Cron ou un scheduler dédié.

## Plan de livraison séquencé

### Semaine 1 — Landing + Paiement Wave
- Page marketing : pitch, screenshots V1 email digest, prix Studio / Premium
- Formulaire d'inscription : nom, agence, email, téléphone
- Flow Wave complet (voir détail ci-dessus)
- Valeur livrée immédiatement : ajout de l'email de l'agence au scraper V1
  (digest quotidien dès J+1, avant même le dashboard)
- **Objectif : 1 agence s'engage à payer (ou paie) avant fin semaine 1**

### Semaine 2-3 — Admin Dashboard (builder-facing)
- Liste des agences : statuts, dates, plan souscrit
- Confirmation Wave en 1 clic + génération facture/reçu PDF
- Stats : nb agences actives, revenu FCFA du mois
- Simple Clerk User auth (pas d'Organizations encore)

### Semaine 3-5 — Agency Dashboard (client-facing)
- Auth Clerk Organizations (création de l'org au moment de l'activation)
- Vue des AOs du jour filtrés (depuis Supabase appels_offres)
  - Filtrage basique : correspondance keyword sur titre et description
  - Interface : titre, source, deadline, lien, statut (nouveau/intéressant/ignoré)
- Inviter des membres selon le plan (Studio: max 2, Premium: max 10)
- Paramètres : keywords supplémentaires, email de notification
- Filtrage avancé (Phase 2) : tags secteur, fourchette budget, date limite

### Phase 2 (post-10 clients actifs)
- Webhook Wave API (quand disponible)
- Filtrage avancé : NEGATIVE_KEYWORDS, fourchette budget, deadline
- Notifications in-app + push browser
- Politique d'offboarding : org suspendue après 7 jours de grâce (non-paiement),
  données retenues 90 jours puis supprimées

## Approches considérées

### Approche A : Email-first Validation
Google Sheet + Tally.so, zéro backend, digest email manuel.
Rejetée : non scalable et sans différenciation produit visible.

### Approche B : Lean SaaS séquencé ← **Retenue**
Landing + Wave semaine 1, dashboard simple semaine 3, Clerk multi-user si demandé.

### Approche C : Full SaaS V1 complet
3 dashboards en parallèle, Clerk multi-tenant dès le départ.
Architecture adoptée, séquence de livraison = Approche B.

---

## Approche recommandée

**B avec architecture C.**

Livraison séquencée pour aller vite, architecture extensible pour ne pas refactorer.
Différence concrète : Clerk et le schéma Supabase multi-tenant sont installés dès
la semaine 1, mais les Organizations ne sont créées qu'en semaine 3 à l'activation.

## Ce que j'ai remarqué dans ta façon de penser

- Tu as dit "B avec architecture prévue pour C" — ce n'est pas un réflexe courant. La plupart des founders choisissent A (trop vite) ou C (trop lent). Tu as trouvé le milieu correct sans qu'on te force.
- Quand on a challengé la PWA, tu as révisé immédiatement vers "responsive only" — sans défendre ton plan original. C'est de la flexibilité fondée sur des preuves, pas de la capitulation.
- Tu as déjà livré V1 du scraper. Ce n'est pas une idée — c'est un produit qui tourne. Tu pars de 50% fait, pas de zéro.
- L'attention au détail du système Wave (le regex T_[A-Z0-9]{18}, les 1% de frais, les numéros de facture séquentiels FAC/REC) — c'est du soin produit, pas de l'over-engineering.
